# queue-service (СКАФОЛД — перейменувати під реальний сервіс)

Приклад сервісу, у якого спільне з `app` — тільки акаунти (через
account-service) і Redis. Власної бізнес-БД тут мінімум (`task_log` —
просто приклад, замінити на реальну схему).

## Структура

```
queue_service/
├── account_port/          ← копія порту до account-service (не пакет, copy-paste)
│   ├── account_client.py    "зроби цей запит"
│   └── account_events.py    "калбек для socket" (Redis pub/sub)
├── config.py
├── db.py                  ← власна SQLite цього сервісу (не accounts/sessions!)
├── worker.py               приклад: слухає власну чергу Redis + реагує на socket-події
└── main.py
```

## Чому account_port скопійований, а не імпортований як пакет

Свідомо: кожен сервіс лишається незалежним і білдиться окремим Docker-образом
без спільного requirements/monorepo-tooling. Якщо сервісів стане багато і
підтримувати N копій набридне — винести `account_port/` у власний пакет
(`pip install account-port` з внутрішнього PyPI/git) і підключати як
залежність. Поки що (2 сервіси) copy-paste дешевший за інфраструктуру пакування.

## Задачі на ті самі акаунти, що й app

`account_id` — той самий, що використовує `app`. Приклад постановки задачі
в чергу (з будь-якого місця, що має доступ до Redis):

```bash
docker compose exec redis redis-cli lpush queue_service:tasks \
  '{"account_id":"acc1","action":"ping_profile"}'
```

⚠ **Важливо** (див. ARCHITECTURE.md → "Кілька сервісів на одному акаунті"):
account-service серіалізує room-switch+запит під локом на акаунт, тож двом
сервісам на одному акаунті одночасно безпечно — просто чекають один одного
в черзі, дані не переплутаються. Але це додаткова затримка при високій
конкуренції за один і той самий акаунт — тримати в голові при масштабуванні.
